## 0. 任务理解修正（重要）

**错误理解（已废弃）：** 重构目标 = 封装成标准 MCP 对外交付。

**正确理解：**

- 现有 `llm-wiki` **已经支持标准 MCP**（`/mcp` + `ask_llm_wiki`），MCP 是出口之一，不是本次重构的主目标。
- 本次重构要回答的核心问题是：  
  **能否用更好的知识库方案 / 检索方案 / 代码仓库扫描方案，把三仓库知识问答做得更准、更全、更稳，同时保留调试 vs 对外可读双模式，且不泄源码/隐私。**

MCP 仍是对外对接形态之一，但方案选型应以 **问答质量与知识管线** 为中心，而不是协议封装。

## 1. 市面上代码仓库「知识库检索」通常怎么做？

业界大致收敛成 **三条哲学 + 一层产品知识**，成熟产品几乎都是混合。

### 1.1 三条检索哲学

| 路线 | 代表 | 做法 | 擅长 | 短板 |
|------|------|------|------|------|
| **Index-first（先索引再答）** | Cursor、Augment、Copilot Enterprise、早期 Cody | 预建 embedding / 符号索引，问答时 top-k 取片段喂给模型 | 语义相似、快速定位相关文件 | 清单/架构类易漏；索引过期；跨仓弱 |
| **Agentic search（现场探索）** | Claude Code 等 | 少预索引，给 grep/read 工具，模型边搜边读 | 灵活、 freshest | token 贵、不稳定、清单易漏/飘 |
| **Graph-augmented（结构图增强）** | Sourcegraph SCIP/Cody、Codebase-Memory (CBM)、各类 code graph | AST/LSP 抽符号与边，图查询 + 可选 embedding | 调用链、影响面、服务/模块结构 | 「产品语义」不全在图里；需维护索引 |

2026 共识（研究综述口径）：**最优往往是混合**——embedding 管相似，graph 管关系，agent 管任务特化探索。

### 1.2 常见技术栈分层（从底到顶）

```
L0 源码 / 文档 / 部署清单
L1 扫描解析：tree-sitter / LSP / SCIP / OpenAPI / K8s / docker-compose
L2 索引存储：向量库 + 符号/图库（SQLite/Neo4j）+ 全文搜索
L3 检索路由：按题型选 lexical / semantic / structural
L4 Agent 编排：多轮 tool + budget + 证据约束
L5 产品知识层（可选但关键）：人工/半自动「权威卡片」——服务清单、能力说明、FAQ
L6 出口：IDE Chat / MCP / API；对外再做脱敏与可读改写
```

**关键洞察：**  
纯「代码 RAG」擅长帮工程师改代码；**产品级知识问答**（「有哪些微服务」「功能清单」）往往还要 **L5 权威知识层**，否则完整率/稳定性上不去——因为「服务边界」不一定等于「目录名」或「图节点」。

### 1.3 各层典型方案举例

| 层 | 市面常见做法 |
|----|----------------|
| 扫描 | tree-sitter 切块；SCIP/LSIF 符号；OpenAPI/proto 抽 API；compose/helm 抽部署单元 |
| 向量检索 | 按函数/文件/AST chunk embedding；rerank |
| 结构检索 | call graph、import graph、route 表、社区聚类 |
| 全文 | ripgrep / Zoekt 类代码搜索 |
| 知识沉淀 | 对话沉淀卡片、文档 wiki、ADR、服务目录（Service Catalog） |
| 安全 | 路径沙箱、secret 扫描、对外答案降敏（你们已有 P0 方向） |

### 1.4 和「清单类问题」的关系（对应你的痛点）

问「chatkit-middleware 有哪些微服务」时：

| 只靠 | 典型失败 |
|------|----------|
| 纯 embedding | 召回若干 README 片段，模型猜「大概 10 个」 |
| 纯 agentic grep | 有时扫到 `services/`，有时漏 submodule / 网关静态挂载 |
| 纯 code graph | 可能列出 Package/Folder，但「产品意义上的微服务」需业务定义 |
| **权威服务目录 + 图/扫描校验** | 清单稳定；扫描负责发现漂移并告警 |

所以市面「做得好的代码知识问答」在清单/架构题上，往往不是更猛的 embedding，而是 **可验证的权威清单 + 索引校验漂移**。

## 2. 对照：llm-wiki 现在站在哪

| 市面层 | llm-wiki 现状 | 评价 |
|--------|---------------|------|
| Agentic 工具 | glob / search_content / read_file | 有 |
| Graph | CBM（`cbm_search`） | 已选结构检索主路径 |
| 检索路由 | plan + hard router + budget | 有 |
| 知识卡片 | P3 大部分完成（fast path、去重） | 有，但未必覆盖「权威服务清单」 |
| 安全 / profile | P0 + debug/public | 有 |
| MCP 出口 | 已有 | **不是缺口** |
| 文档/wiki 摄入 | P6 未做 | 缺口 |
| 多源融合引擎 | P4 未做 | 缺口 |
| 清单 ground truth | 未显式产品化 | **与目标 1 最相关的缺口** |

结论：重构焦点应是 **知识管线与检索质量**，不是 MCP 协议本身。

## 3. 「更好的方案」可选方向（供后续决策，此处不定案）

| 方向 | 含义 | 可能收益 | 成本 |
|------|------|----------|------|
| D1 强化权威知识层 | 服务/功能/模块清单卡片 + 版本 hash + 漂移检测 | 完整率、稳定性 | 需定义 ground truth 与更新流程 |
| D2 加强扫描源 | 除 CBM 外，固定解析 docker-compose / 服务目录 / OpenAPI | 清单类准确 | 规则维护 |
| D3 换/补检索后端 | 评估 Zoekt、SCIP、其他 graph、更好 embedding | 视题型 | 集成与运维 |
| D4 题型专用管线 | listing/architecture 走确定性抽取，禁止自由发挥 | 稳 | 覆盖面要设计 |
| D5 评测驱动 | 扩大 golden，清单题用集合 F1 判分 | 可证明变好 | 标注成本 |

**YAGNI 提醒：** 不必先换整套引擎；先用评测证明瓶颈在「缺权威清单」还是「CBM/检索不够」。

## 4. 对 kickoff 文档的修正摘要

- 主目标从「MCP 封装」改为「更好的知识库 / 检索 / 扫描 → 更好的知识问答」。
- A1（知识策略）仍然关键，但语境变为：**在已有 MCP 出口前提下，知识从哪来才撑得住准确/完整/稳定。**
- F2（MCP 传输形态）降为次要，除非对接方有新要求。
