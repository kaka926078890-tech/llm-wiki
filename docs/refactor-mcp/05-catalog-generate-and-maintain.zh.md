# 权威名单：如何生成、如何维护

日期：2026-07-10  
状态：讨论草案（待确认口径后可进 decision）  
关联：[04 白话什么是名单](./04-what-is-catalog-plain.zh.md)

---

## 0. 一句话

| 环节 | 怎么做 |
|------|--------|
| **生成** | 脚本从仓库目录/部署文件扫出「候选名单」→ 人审改成「权威名单」 |
| **维护** | 代码变了再扫一遍，和权威表 diff；有新增/删除就改表并记一笔；问答始终读权威表 |

**机器提候选，人定对错；问答不现场发明名单。**

---

## 1. 生成：两步，不要一步到位

### 第 1 步：自动生成「候选名单」（Candidate）

脚本扫仓库，列出「看起来像服务」的目录，例如对 `chatkit-middleware`：

| 扫描源 | 扫什么 | 得到什么 |
|--------|--------|----------|
| A. 目录约定 | `services/gateways/*`、`services/platform/*`、`services/pulse/*`、`services/enterprise/*` | 路径 + 文件夹名 |
| B. 部署文件（可选） | `docker-compose*.yml`、`edition-manifest.yaml` 里的服务名 | 真正会部署的名字 |
| C. 包特征（可选） | 目录下有 `package.json` / `Cargo.toml` / `Dockerfile` | 更像可部署单元 |
| D. CBM（可选） | architecture/packages 信号 | 仅作补充，**不能单独当权威** |

输出示例（候选，还不能直接当答案）：

```text
gateways/ag-ui-server
gateways/api-gateway
platform/identity-service
platform/orchestrator
…
```

这一步 **可以全自动、可重复跑**，每次得到「代码眼里现在有什么」。

### 第 2 步：人审变成「权威名单」（Authoritative）

人对着候选表做四件事（写在 overlay / 权威 JSON 里）：

1. **留 / 删**：这个算不算「微服务」？（例如 tools、纯库、demo 可能删掉）  
2. **改名**：对外显示名（`identity-service` →「身份认证服务」）  
3. **写一句话**：非技术人员能懂的职责  
4. **定口径说明**：表头写清「本次清单包含/不包含什么」

审完落盘，例如：

```text
config/catalog/chatkit-middleware.services.yaml   # 权威表（人维护）
.reasonix/catalog-candidates/….json               # 最近一次扫描候选（机器生成，可覆盖）
```

**只有权威表能进问答答案。** 候选表只给维护的人看。

### 第一次怎么冷启动？

```
1. 跑一次扫描 → 得到候选列表（约几十行）
2. 你（或熟悉系统的人）花 30–90 分钟勾选、改名、写一句话
3. 存成 v1 权威表
4. 用这张表跑 E0/E1 对比（验证名单题是否变稳）
```

不需要先上复杂平台；**第一次就是「扫描 + 人工勾选」**。

## 2. 维护：日常谁改、何时改

### 2.1 触发时机

| 何时 | 做什么 |
|------|--------|
| 合并了会增删服务的 PR | 再跑扫描，看 drift |
| 定期（如每周 / 每次 `sync:code`） | 扫描 vs 权威表 diff |
| 问答答错/漏了 | 改权威表（修内容），不是先改模型 |

### 2.2 Drift（漂移）报告长什么样

```text
【新增候选，权威表没有】
  + services/platform/new-foo-service

【权威表有，代码里没了】
  - services/platform/old-bar-service

【未变化】
  27 项
```

维护者看到报告后：

- 确认新服务 → **写入权威表**（名字 + 一句话）  
- 确认已下线 → **从权威表删除**  
- 误报（不算微服务）→ 写入 overlay 的 **排除列表**，下次扫描不再烦你

### 2.3 维护分工（建议）

| 角色 | 职责 |
|------|------|
| 脚本 / CI | 扫描、出 drift、挡住「权威表严重过期」可选 |
| 熟悉三仓的人（你们） | 审 drift、改权威表、定口径 |
| llm-wiki 问答 | **只读权威表** 回答名单题；不擅自加服务名 |

一期可以没有 CI，本地 `npm run catalog:scan` + 人工看 diff 就够。

### 2.4 版本怎么管

权威表跟 git 走：

- 改表 = 一次普通 commit  
- 表里可记 `commit` / `updatedAt`，方便回答「这份知识对应哪版代码」  
- 代码已变、表未更新 → 可在答案里提示「名单可能未含最近新增服务」（可选）

## 3. 推荐的最小流程（YAGNI）

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ 扫仓库       │ ──► │ 人审 overlay │ ──► │ 权威 JSON   │
│ → 候选列表   │     │ 留删改名写摘要│     │ （git 管理） │
└─────────────┘     └──────────────┘     └──────┬──────┘
       ▲                                        │
       │              定期/发版再扫               │
       └──────── drift 报告 ◄───────────────────┘
                                                │
                                                ▼
                                         名单题直接读表回答
```

**不做（一期）：** Backstage、独立微服务注册中心、让 AI 自动写权威表并直接上线。

**AI 可以帮忙：** 根据 README 草拟「一句话职责」，但 **必须人点确认** 才进权威表。

## 4. 和「生成答案」的关系（避免再混）

| | 名单表 | 问答时 |
|--|--------|--------|
| 生成 | 扫描 + 人审，**事先**完成 | 不生成名单，只 **读取** |
| 维护 | 人改表 / 处理 drift | 发现漏了 → 开 issue 改表 |

所以：  
「名单如何生成」≠「每次回答时生成」；  
是「发版/变更时生成并维护好，回答时照念」。
