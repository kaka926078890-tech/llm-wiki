# 「权威名单表」是什么（白话版）

日期：2026-07-10  
状态：概念澄清  
关联：[03 Catalog 与验证](./03-discussion-catalog-and-validation.zh.md)

---

## 一句话

**权威名单表 = 你们事先写好、并约定「这就是正确答案」的一张服务名单。**

不是数据库产品，不是检索引擎，不是 MCP。  
就是一份文件（JSON / YAML / 甚至 Excel 导出），里面列着：

> chatkit-middleware 里，我们承认有哪些服务，每个服务一句话干什么。

以后有人问「有哪些微服务」，系统 **照着这张表念**，而不是让 AI 临时去代码里数一遍。

---

## 用生活类比

问前台：「咱们公司有几个部门？」

| 做法 | 结果 |
|------|------|
| 每次让新人在大楼里乱逛再猜 | 有时说 10 个，有时漏几个，每次不一样 |
| 前台抽屉里有一张 **人事认定的部门名单** | 每次都一样，漏了就改名单，不改「猜法」 |

**权威名单表 = 那张人事认定的部门名单。**  
Catalog 只是给这张表起的技术名字。

---

## 具体长什么样（假例子）

文件可以叫：`benchmarks/catalogs/chatkit-middleware.services.json`

```json
{
  "repo": "chatkit-middleware",
  "list_name": "微服务清单",
  "note": "口径：services 下可独立部署的服务；不含 libs/tools",
  "services": [
    {
      "name": "identity-service",
      "summary": "管登录、注册和谁是管理员"
    },
    {
      "name": "ag-ui-server",
      "summary": "管聊天对话的实时推送"
    },
    {
      "name": "orchestrator",
      "summary": "按流程编排各个 AI 步骤"
    }
  ]
}
```

上面只有 3 条是示意。真正的表会有你们认定的全部服务（比如 20～40 个），**每一条都是人点过头的**。

问：「chatkit-middleware 有哪些微服务？」

- **现在**：AI 搜代码 → 自己总结 → 可能说「一共 10 个」→ 错/漏/每次不同  
- **有表之后**：系统读这张表 → 把 `services` 列出来 → 再用人话写两句 → **名单以表为准**

---

## 它解决什么、不解决什么

| 解决 | 不解决 |
|------|--------|
| 「有哪些 / 共几个」这类 **名单题** 的对错、漏项、飘忽 | 「某函数谁调用了」——那仍靠代码搜索 / CBM |
| 给非技术人员稳定、可读的清单 | 自动理解所有新代码细节（新服务仍要 **更新这张表**） |

---

## 谁维护这张表？

1. **人定口径**：什么叫微服务（例如：只算 `services/` 下可部署的，不算工具脚本）。  
2. **人审名单**：第一次把名单写对。  
3. **可选**：脚本扫描目录，提示「代码里多了一个目录，表里没有」——提醒你改表，**不是让 AI 当场发明名单**。

没有「人审过的名单」，就没有权威；扫描只是助手。

---

## 和「知识卡片」差在哪？

| | 知识卡片 | 权威名单表 |
|--|----------|------------|
| 像什么 | 一篇问答笔记 | 通讯录 / 花名册 |
| 问「有哪些服务」 | 看笔记写全没写全，问法一变可能找不到 | 直接打开花名册，按名单念 |
| 能不能验「漏没漏」 | 很难自动验 | 很容易：答案集合 vs 表里集合 |

---

## 你需要记住的唯一定义

> **权威名单表 = 事先写好的正确答案名单（带一句话说明）。**  
> Catalog = 这张表在方案里的称呼。  
> 目的 = 名单题别再让模型现场猜。

若这张表你觉得「就是人工维护一份 JSON」，理解就对了——一期本来就可以这么简单。
