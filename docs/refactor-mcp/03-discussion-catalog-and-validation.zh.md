## 0. 先给结论

| 问题 | 短答 |
|------|------|
| Catalog 是什么？ | **一张带版本的「权威名单表」**：系统里有哪些服务/模块、各干什么、算不算进某类清单。不是新框架，更像 Excel/YAML 真相源。 |
| 怎么证明决策让回复更好？ | **先跑基线，再改，再比数。** 答不好就回滚。没有分数提升 = 决策失败。 |
| 如何不是技术堆砌？ | 每项改造必须对应一个 **可失败的质量假设**；过不了假设的组件不做/删掉。 |
| 每次改造怎么验？ | **一条改造 = 一条假设 + 一组前后对比题 + 通过线**；进主路径前必须绿。 |

## 1. Catalog 到底是什么内容？

### 1.1 一句话

**Catalog = 产品口径下的实体名录 + 非技术说明 + 证据指针。**

它解决的不是「怎么检索」，而是：

> 「微服务清单」这类问题，**答案集合在提问前就应该已经存在**，模型只负责按口径取出并说人话——而不是每次从代码里重新发明名单。

### 1.2 里面有什么（最小字段）

以 `chatkit-middleware` 为例，一条记录长这样：

```yaml
id: chatkit-middleware/identity-service
repo: chatkit-middleware
kind: service                    # service | gateway | spa | library | worker …
name: 身份认证服务
summary: 负责登录、注册、租户与管理员权限判断。   # 非技术人员能懂
capabilities:
  - 邮箱/用户名登录与注册
  - 平台管理员白名单
  - 租户管理员标记
paths:                           # 仅 debug 可见；public 默认不输出
  - services/platform/identity-service
evidence:
  - path: services/platform/identity-service/README.md
    hash: "…"
include_in:
  microservice_list: true        # 产品口径：算不算「微服务清单」
commit: abc1234
```

整张 Catalog 就是 **几十到一两百条这样的记录**（三仓合计），外加一份 **overlay 规则**（哪些目录算、哪些不算、如何改名）。

### 1.3 它不是什么

| 不是 | 说明 |
|------|------|
| 不是向量数据库 | 不靠 embedding 找「像不像」 |
| 不是 CBM 替代品 | CBM 仍管调用链/结构；Catalog 管「名单与口径」 |
| 不是 Backstage 整站 | 一期可以是一个 JSON/YAML + 几十行查询代码 |
| 不是让模型更聪明 | 是让模型 **没机会猜错名单** |

### 1.4 和现有「知识卡片」的差别

| | 知识卡片（现状） | Catalog（提议） |
|--|------------------|-----------------|
| 形态 | 自由文本 Q&A | 结构化实体表 |
| 匹配 | 问句像不像（lexical） | 题型=清单 → 直接查表 |
| 完整率 | 靠人写全、难验集合 | 集合可算 F1 |
| 稳定性 | 问法一变可能 miss | 同口径同集合 |

**若一期只做一件事：** 把「middleware 微服务名单」做成一份 JSON，listing 题直接渲染——这已经是 Catalog MVP，不必先上「扫描框架」。

### 1.5 谁往里写内容？

| 来源 | 作用 |
|------|------|
| 扫描候选 | 从 `services/**` 等扫出「可能是服务」的目录 |
| Overlay（人审） | 决定算不算微服务、显示名、一句话职责 |
| README/文档（可选） | 补 `summary` / `capabilities` 文案 |
| Drift 报告 | 代码新增了目录但表里没有 → 告警，而不是静默漏答 |

**内容质量 = 人审口径 + 扫描防漏。** 没有人审口径，扫描再强也会答错「什么叫微服务」。

## 2. 如何验证「这些决策会让回复更好」？

### 2.1 核心原则：可证伪，不讲故事

每个架构决策写成假设：

```
假设 H：若 listing 题改为「读 Catalog 再表达」，
则 同一组清单题 的集合 F1 与同问稳定性 相对基线显著提升，
且 public 泄密 lint 不恶化。
若提升不达标 → 假设失败，回滚该路径，不继续堆 Fusion/Embedding。
```

**没有假设、没有前后对比 = 不允许合入主路径。**

### 2.2 最小验证实验（建议 Phase 0 就做，甚至先于写 Catalog 代码）

**实验 E0（证明痛点可测）：**

1. 人工定一份「middleware 微服务权威集合」S*（哪怕先 20–30 个名字，写进 `benchmarks/catalogs/…json`）。  
2. 用 **当前** `ask_llm_wiki` 对固定问句跑 N=3 次。  
3. 从答案抽「服务名集合」S_i，算：
   - Precision / Recall / F1（相对 S*）
   - 稳定性：三次两两 Jaccard  
4. 记下耗时、是否出现「共 N 个」且 N≠|S*|。

**实验 E1（证明 Catalog 有效）：**

1. 实现「读 S* 模板输出」（可以先硬编码，甚至脚本生成答案）。  
2. 同一问句再跑 N=3。  
3. 对比 E0：F1、Jaccard、错误计数是否提升。

| 结果 | 含义 |
|------|------|
| E1 明显好于 E0 | Catalog 方向成立，值得工程化扫描/overlay |
| E1 ≈ E0 | 问题不在名单，可能在抽名/口径/问法；先别堆架构 |
| E1 更好但 public 泄密变差 | 投影规则有问题，修 renderer，不否定 Catalog |

**这就是「验证决策」：用 E0→E1 的数字说话，而不是用市面调研说话。**

### 2.3 决策 → 假设 → 度量 对照表

| 决策 | 质量假设（可失败） | 主度量 | 失败则怎么办 |
|------|-------------------|--------|--------------|
| D1 Catalog-first | 清单题完整率/稳定性升 | listing F1、Jaccard | 停；分析是口径错还是管线错 |
| D2 保留 CBM | 符号/调用题不退化 | golden 中 symbol/arch 子集 | 修 cbm 用法，不急着换引擎 |
| D3 确定性 listing | 不再出现错误「共 N 个」 | 计数违规=0；F1≥阈值 | 回退 Agent listing |
| D4 ABCD 全要 | 有基线可对比 | 报告存在且可复现 | 先补评测再改代码 |
| D5 不引入重平台 | 同等质量更少依赖 | 依赖数、运维步骤 | 若自研 Catalog 维护不动再评估 |
| D6 Cards→FAQ | 清单不再依赖卡片命中 | listing 不走错误 fast-path | 收紧 fast-path 题型 |

市面调研只解释 **为什么值得做 E1**；**是否采纳** 只看 E1 是否赢。

## 3. 判断标准：怎样算「不是技术框架堆砌」？

### 3.1 准入三问（任一答「否」→ 不做）

对每一个拟议组件（Fusion、Embedding、Backstage、新图数据库…）问：

1. **它直接改善哪条用户可见失败？**（错 / 漏 / 飘 / 泄密 / 难懂）  
2. **有没有比它更小的改动可能达到同样效果？**（例如一张 JSON + 模板）  
3. **两周内能否用前后对比数字证明有效或无效？**

### 3.2 堆砌红灯（出现即停）

- 为对齐「市面架构图」而加层，但 **没有对应失败用例**。  
- 新组件上线后 **主度量不变**，只多了配置/运维。  
- 同时开多条大改（Catalog + 换引擎 + 新 UI），无法归因。  
- 「先搭好平台再填内容」——内容（权威名单）才是清单题的瓶颈。

### 3.3 绿灯（允许做）

- 改动面小，但打中高频失败（如 listing 猜数）。  
- 先 **内容/口径**（S* + overlay），再自动化扫描。  
- 每阶段只有 **一个主变量**（先 Catalog 路径，再谈 Fusion）。  
- 失败可回滚：feature flag 切回旧 Agent listing。

### 3.4 对「Catalog」本身的诚实定位

Catalog **可以**沦为堆砌——如果做成：

- 空壳服务注册中心、复杂 schema、没人维护的 overlay、答案仍让模型自由发挥名单。

Catalog **不是**堆砌——如果做成：

- 一份人审过的名单 + 确定性输出 + F1 门禁；扫描只是减少漏维护。

**判断标准：用户问清单时，答案集合是否等于那张表（在口径内）。** 等于 → 有效；不等于 → 无效，与用了多少「架构名词」无关。

## 4. 每次改造如何验证？（操作手册）

### 4.1 改造工单模板（建议强制）

```markdown
## 改造标题
假设：…
影响题型：listing | architecture | …
主度量：F1 / Jaccard / golden 子集 / public lint / 人工分
基线：commit / 报告路径 / 分数
改动：文件列表（尽量小）
对比题：固定 3–10 问（含你的反例）
通过线：例如 listing F1 ≥ 0.95 且 Jaccard ≥ 0.95 且 lint 不恶化
回滚：如何关掉新路径
```

### 4.2 推荐验证流水线

```
1. 冻结对比题集 Q（含「middleware 微服务清单」）
2. 跑 baseline → benchmarks/reports/baseline-*.json
3. 开 feature flag 合入改造
4. 跑 candidate → benchmarks/reports/candidate-*.json
5. diff 报告：A/B/C 相关指标
6. 绿 → 合入；红 → 回滚或缩小范围
7. 人工抽检 C（非技术 2–3 人看 public 答案）— 重大路径必做
```

### 4.3 分期验证焦点（防一次堆完）

| 阶段 | 只验证什么 | 先不做 |
|------|------------|--------|
| Phase 0 | 权威集合 S* 存在；E0 基线可复现 | 任何新框架 |
| Phase 1 | E1：Catalog/模板路径打赢 E0 | Fusion、Embedding、新 UI |
| Phase 2 | CI 门禁防回退；drift 能抓住漏项 | 换 CBM |
| Phase 3 | 非清单题是否也升（才考虑 fusion） | 重型平台 |

### 4.4 「变好」的操作定义（建议写死）

对 **清单题子集**：

- **准确**：不出现错误总数；名称在口径内无张冠李戴。  
- **完整**：Recall ≥ 阈值（建议 0.95）。  
- **稳定**：N=3 次答案集合 Jaccard ≥ 0.95。  
- **对外**：public 答案无源码块/密钥/内部 URL；非技术可读抽检通过。

对 **非清单题**：至少 **不显著变差**（golden 子集通过率 ≥ 基线 − 容差）。

## 5. 直接回答你的四句质疑

**Q: Catalog 是什么内容？**  
A: 人审过的「实体名单 + 一句话职责 + 是否计入某类清单」；可先是一个 JSON。扫描/CBM 只是帮你发现该更新名单。

**Q: 如何验证决策让回复更好？**  
A: E0 基线 vs E1 Catalog 路径，比 F1/Jaccard/猜数错误；赢了才工程化，输了就停。

**Q: 确定不是技术堆砌？判断标准？**  
A: 三问准入 + 红灯停工；主标准是 **用户可见失败是否减少**，不是架构图是否漂亮。Catalog 若只是多一层而集合仍靠模型猜，那就是堆砌，应删。

**Q: 如何验证每次改造？**  
A: 一改造一假设一对比报告；feature flag；红则回滚；阶段上一次只动一个主变量。

## 6. 建议的下一步（仍只确认一点）

在写更多架构代码之前，先做 **E0 基线**（可不改产品代码）：

1. 你确认「微服务」口径草案（或授权我用 `services/**` 可部署单元出一版供你删改）。  
2. 固化 1 道反例题 + 抽名规则。  
3. 跑 3 次现网 MCP 答案，产出基线数字。

有了基线，后面任何「架构决策」都有裁判；没有基线，讨论容易滑向框架堆砌。
