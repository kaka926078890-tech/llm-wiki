# 功能抽取发生在哪一步？

日期：2026-07-10  
状态：流程澄清  
关联：[08 功能清单如何生成](./08-how-feature-list-is-generated.zh.md)

---

## 一句话

**抽取不在用户提问时做。**  
它放在 **代码同步 / 索引更新** 这一步（和 `sync:code`、`cbm:sync` 同级），事先生成 `feature-lists/*.json`；  
用户问「功能清单」时，问答步骤 **只读这份 JSON**。

---

## 整条时间线（相对现有命令）

```
① 拉代码          npm run sync:code
                      │
② 更新结构索引     npm run cbm:sync          ← 已有（CBM 图）
                      │
③ 抽取功能清单     npm run catalog:gen       ← 新增（建议挂在 sync:code:full 里）
                      │                         写出 .reasonix/feature-lists/{repo}.json
                      ▼
                 【平时待命】

④ 用户提问         /agent/run 或 MCP ask_llm_wiki
                      │
                   若是「功能清单」类问题
                      │
                   ⑤ 读 JSON 渲染答案          ← 不跑抽取器、不扫全仓编名单
                      │
                   其他问题仍走 CBM / 读文件等
```

| 步骤 | 何时 | 做什么 | 现在有没有 |
|------|------|--------|------------|
| ① sync:code | 代码更新时 | clone/pull 三仓 | 有 |
| ② cbm:sync | 同上或随后 | 重建/增量 CBM 图 | 有 |
| **③ catalog:gen** | **同上或随后** | **跑抽取器 → 功能清单 JSON** | **方案新增** |
| ④ 提问 | 用户随时 | Agent/MCP 入口 | 有 |
| **⑤ 读表回答** | **清单类问题** | **读 JSON，不现场抽取** | **方案新增** |

建议：`npm run sync:code:full` = ① + ② + ③（一次更新知识侧全部产物）。

---

## 明确：不在哪些步骤做抽取

| 步骤 | 抽不抽 | 原因 |
|------|--------|------|
| 用户每次提问 | **不抽** | 又慢又不稳，会回到「现场猜名单」 |
| LLM 思考过程中 | **不抽** | 模型不应发明条目 |
| CBM 查询内部 | **不抽** | CBM 管结构图，不管产品/页面功能表 |
| 仅前端打开页面时 | **不抽** | 清单是后端知识产物 |

---

## 和现有「知识卡片」步骤的区别

| | 知识卡片（已有） | 功能清单抽取（拟增） |
|--|------------------|----------------------|
| 何时产生 | 人从某次问答「保存」 | 同步时脚本自动生成 |
| 触发 | 手动 | `catalog:gen` / sync:full |
| 清单题是否依赖 | 不该依赖（问法一变易 miss） | **应依赖**（确定性） |

---

## 若只记一个位置

> **抽取 = 离线知识构建步骤（步骤 ③），挂在代码同步之后。**  
> **回答 = 在线步骤（步骤 ⑤），只消费抽取结果。**
