# 三仓并不对称：没有统一「服务清单」时怎么自动化

日期：2026-07-10  
状态：讨论澄清（修正「每个仓都能像 middleware 一样出微服务表」）  
关联：[06 自动化与人审](./06-automated-catalog-without-expert.zh.md)

---

## 0. 你的判断成立

| 事实 | 含义 |
|------|------|
| 文档不齐全 | 不能指望 README 自动生成「完整产品功能清单」 |
| 只有 middleware 有 `edition-manifest` | **不能**把「解析 manifest → 微服务表」当成三仓通用方案 |
| chatkit-web / finclaw 没有同类服务清单 | 它们要列的不是「微服务」，而是 **别的可枚举单元** |

因此：权威名单不是一种表，而是 **「按仓库、按问题类型，选该仓真实存在的可枚举源」**。  
没有源，就诚实说「无法给出完整产品功能清单」，而不是硬编一张假表。

---

## 1. 先改问题：不要三仓都问「微服务」

| 仓库 | 适合自动枚举的「名单」 | 不适合假装有的 |
|------|------------------------|----------------|
| **chatkit-middleware** | 部署服务（manifest / compose） | — |
| **chatkit-web** | 应用/包（workspaces）、页面能力（路由）、集成点（IM 等代码常量） | 「微服务列表」 |
| **finclaw** | Cargo crates / hosts / 能力面（HTTP 路由、skills 机制） | 「微服务列表」；完整「产品功能册」若文档缺失则只能给结构清单 |

用户问「功能清单」时，系统应 **改写成该仓可回答的清单类型**，并在答案里写清口径，例如：

> 以下是 chatkit-web 的 **前端应用与模块清单**（来自 package.json workspaces），不是微服务。

而不是硬套 middleware 的服务表。

---

## 2. 各仓「能自动采信」的源（务实）

### 2.1 chatkit-middleware（有 SSOT）

- **主源：** `edition-manifest.yaml` → 服务名 + path  
- **交叉：** compose  
- **摘要：** README 有则抽，无则只给名字 + 路径（debug）/ 名字（public）

### 2.2 chatkit-web（无服务 manifest，有结构 SSOT）

| 清单类型 | 自动源 | 完整率预期 |
|----------|--------|------------|
| 有哪些应用/包 | 根 `package.json` → `workspaces` | **高**（工程结构即真相） |
| 管理后台有哪些功能区 | `chatkit-admin-mt` 路由表（`App.tsx` / router） | **中高**（有路由≈有入口；文案可能糙） |
| 用户端有哪些主功能 | `finclaw-frontend` 路由 | **中高** |
| IM 频道等 | 代码里的 connector/常量 + 已有 golden | **中高** |
| 「产品级完整功能说明书」 | 文档 | **低**（文档不全就承认不全） |

**web 的自动化 = 结构枚举（workspaces + routes），不是编造产品手册。**

### 2.3 finclaw（Rust 单体/工作区，无微服务表）

| 清单类型 | 自动源 | 完整率预期 |
|----------|--------|------------|
| 有哪些 crates / 模块 | `Cargo.toml` workspace members、`crates/*` | **高** |
| 有哪些 host（CLI/desktop） | `hosts/*` | **高** |
| 对外能力面 | HTTP route / OpenAPI / contract 子模块 | **中** |
| 「Claw 全部产品功能」 | docs | **视文档；不全则结构清单 + 明确缺口** |

---

## 3. 文档不全时的策略（核心）

分清两种「功能清单」：

| 类型 | 含义 | 文档不全时 |
|------|------|------------|
| **S-清单（结构清单）** | 仓库里客观存在的应用/服务/crate/路由 | **可自动化，且应优先答这个** |
| **P-清单（产品功能说明书）** | 面向业务的完整能力描述 | 文档差则 **无法保证完整**；禁止假装完整 |

规则：

1. 能答 S → 答 S，并标明「来自 workspaces/路由/manifest」。  
2. 用户要 P → 用 S 作骨架；有 README 的补一句；**缺文档的条目标「仅有工程入口，缺产品说明」**。  
3. **禁止**在无源时生成「共 N 大功能」这种封闭完整表述。  
4. 评测上：S-清单用集合 F1；P-清单单独标为 weak/optional，不拿来卡「完整率 0.95」除非有权威文案源。

这才是对「文档不齐全」诚实的架构，而不是再堆一个空 Catalog 框架。

---

## 4. 修正后的「权威名单」定义（三仓版）

**权威名单 = 对该仓、该题型，从约定自动源抽出的结构化枚举结果（带来源与置信度）。**

- middleware 服务题 → manifest  
- web「有哪些应用」→ workspaces  
- web「管理台功能」→ routes  
- finclaw「有哪些模块」→ crates  

**不是**「全世界统一的一张微服务 Excel」。  
**也不是**「没有源也要人审出一张假完整功能表」。

---

## 5. 对方案的影响（避免技术堆砌）

| 之前容易滑向 | 现在应落地 |
|--------------|------------|
| 先建通用 Catalog 平台 | 先做 **按仓 adapter**：manifest / workspaces / routes / crates |
| 三仓同一套「微服务」话术 | 题型路由时 **改写清单类型** |
| 文档驱动完整 P-清单 | 默认交付 S-清单；P 为尽力而为 |
| 人不熟就人审全表 | 人不审细节；只定「用哪个自动源」的规则 |

验证仍然用 E0/E1，但 **按清单类型分别建权威集合**：

- middleware：manifest 服务集合  
- web：workspaces 集合、或路由 path 集合  
- finclaw：crate 名集合  

没有集合源的「产品功能散文题」→ 不纳入 A 类硬门槛，或降级为人工抽检 C。

---

## 6. 建议决策（请确认）

**R2：** 接受三仓不对称；名单自动化 = 每仓绑定不同结构源；文档不全时只保证结构清单（S），不承诺产品说明书（P）完整。

**R3：** chatkit-web / finclaw **不做**「假微服务表」；问到微服务时回答「该仓不是微服务架构 / 请改问应用或模块清单」，或自动改写问题。

下一问只确认：**是否采纳 R2 + R3？**
